このシステムについて

ワンタッチ管理システムは、介護施設の設備トラブルを「ワンタッチ」で通報し、担当管理会社に自動で連絡が行く仕組みです。

登場人物

  • システム管理者（あなた） — システム全体を管理。管理会社登録、会社セットアップ、契約管理を行います
  • 施設スタッフ — 設備の故障を発見したら通報します。それだけ。簡単です
  • 事業所管理者 — 自分の事業所の通報状況を確認し、対応を管理します
  • 本社管理者 — 全事業所の通報を横断的に見れます
  • 管理会社 — 通報を受け取り、修理・対応を行います

商品のカテゴリと契約設定に基づいて、担当管理会社が自動で決まります。契約が設定されていない場合は、商品マスタで手動で管理会社を選択できます。

契約テーブルとは

CSVインポート時に「担当管理会社」列を省略できるようにするための補助機能です。カテゴリごとに管理会社を紐づけておくと、インポート時に自動で管理会社が設定されます。

主な用途

  • インポートの省力化 — CSVに管理会社列がなくても、カテゴリから自動で管理会社を解決
  • 通報時の自動振り分け — 通報された商品のカテゴリから担当管理会社を特定して通知

例

  • 「建物・外」カテゴリ → TAMJ建設 に割り当て済みの場合、该当カテゴリの商品インポート時に自動セット
  • 契約未設定のカテゴリは、商品マスタで手動で管理会社を選択できます
契約は事業所単位でも設定できます。「全事業所」にすれば、その会社の全施設に適用されます。

日常運用でやること

毎日（推奨）

  • 通報履歴を確認 — 未対応の通報がないかチェック。担当未割当や長期未対応がないか確認
  • 管理会社のステータス確認 — 「対応中」のまま止まっている案件がないか確認

週1回（推奨）

  • 監査ログ確認 — 不審な操作がないか確認（システム設定 → 監査ログ）
  • データバックアップ — システム設定 → データエクスポートで主要データをCSV出力

月1回（推奨）

  • アカウント棚卸 — 退職したスタッフのアカウントを無効化
  • 管理会社の対応実績確認 — 管理会社ダッシュボードで対応件数・完了率をチェック

よくある運用シナリオ

スタッフが退職した場合

  1. マスタ管理 → アカウントマスタを開く
  2. 該当スタッフを検索して「編集」
  3. アカウントを削除、または初期パスワードにリセットして引き継ぎ

新しい事業所を追加する場合

  1. マスタ管理 → 事業所マスタから「新規追加」
  2. 事業所名・住所・電話番号を入力
  3. 契約管理画面で、その事業所のカテゴリ別担当管理会社を設定
  4. アカウントマスタでスタッフ・管理者アカウントを発行
「全事業所」の契約がある場合は、新しい事業所にも自動で適用されます。個別の契約が必要な場合だけ追加設定してください。

管理会社が変わった場合

  1. 新しい管理会社を管理会社マスタで登録（未登録の場合）
  2. 契約管理画面で旧管理会社の契約を「停止」にする
  3. 新しい管理会社の契約を作成（カテゴリを紐づけ）
  4. 必要に応じて、商品マスタで該当商品の担当管理会社を手動更新
契約を更新すると、以降のインポートや通報では新しい管理会社が自動で適用されます。既存の商品の担当管理会社は商品マスタから個別に変更できます。

商品を大量に追加する場合

  1. Excelファイルに商品情報を入力（商品名、メーカー、型番、カテゴリ等）
  2. 商品マスタ → インポートからファイルを取り込み
  3. 列の割当を確認し、プレビューで内容チェック
  4. 問題なければ「取り込み」を実行

パスワードを忘れたスタッフがいる場合

  1. マスタ管理 → アカウントマスタを開く
  2. 該当アカウントの「編集」→「初期パスワードにリセット」
  3. リセット後、スタッフに初期パスワードを伝える
  4. 次回ログイン時にパスワード変更推奨バナーが表示される

トラブルシューティング

通報したのに管理会社に通知が行かない

  • 商品マスタで該当商品の「担当管理会社」が設定されているか確認
  • 商品のカテゴリが正しく設定されているか確認
  • 管理会社のアカウントが有効か確認
  • 契約管理画面で該当カテゴリの契約が「有効」になっているか確認（インポートで自動設定した場合）

管理会社がダッシュボードにログインできない

  • ログイン画面で会社コードに「管理会社コード」（例: PN001）を入力しているか確認
  • 管理会社マスタでログインIDとパスワードが正しいか確認
  • 必要に応じてパスワードを再設定

商品インポートでエラーが出る

  • Excelファイルの1行目が列名（ヘッダー）になっているか確認
  • 必須項目（商品名）が空でないか確認
  • CSV形式の場合、文字コードがUTF-8かShift-JISか確認

画面が正しく表示されない

  • ブラウザのキャッシュをクリア（Ctrl+Shift+R / Cmd+Shift+R）
  • 別のブラウザで試す
  • スマートフォンの場合、ホーム画面のアイコンを削除して再追加

ログインの仕組み

全ユーザーは3つの情報でログインします。

  • 会社コード — 施設スタッフなら会社コード（例: TAMJ）、管理会社なら管理会社コード（例: PN001）
  • ログインID — 「会社コード＋ログイン名」が自動生成（例: TAMJ-yamada）。アカウント登録時にQRコードも発行されます
  • パスワード — 初回は管理者が設定したパスワード。変更可能。初回ログイン時に変更を推奨するバナーが表示されます

よくある質問

管理会社が変わったらどうする？
契約管理画面で旧管理会社の契約を「停止」にして、新しい管理会社の契約を作成します。以降のインポートや通報では新しい管理会社が自動で適用されます。既存の商品の担当管理会社は商品マスタから手動で変更できます。
新しい事業所を追加したい
マスタ管理 → 事業所マスタから追加できます。追加後、契約設定でその事業所のカテゴリ別担当管理会社を設定してください。「全事業所」の契約がある場合は自動的に適用されます。
スタッフのアカウントを増やしたい
マスタ管理 → アカウントマスタから追加できます。本社管理者は自社のアカウント、事業所管理者は自事業所のアカウントを追加できます。
管理会社に複数の担当者がいる場合は？
管理会社マスタの編集画面で「＋担当者を追加」から追加できます。各担当者に個別のログインIDが発行され、それぞれ管理会社ダッシュボードにログインできます。
パスワードを忘れた場合は？
システム管理者がアカウントマスタからパスワードをリセットできます。施設の管理者もアカウントマスタのパスワード変更機能で再設定できます。
商品が多すぎて1件ずつ登録するのが大変
商品マスタのインポート機能を使ってください。ExcelやCSVファイルから一括で登録できます。列の自動マッピング機能もあります。
「その他」カテゴリの通報はどうなる？
「その他」カテゴリには通常、管理会社が割り当てられていません。事業所管理者に通知が行き、施設側で対応を判断します。購入品（洗濯機など）の故障報告などに使います。
DEMOモードとは？
ログイン画面下部の「体験」ボタンからアクセスできます。サンプルデータで各役割（スタッフ、管理会社、管理者）の画面を体験できます。DEMOのデータは本番データに影響しません。
データを全て消したい
システム設定画面の「全データ削除」ボタンから実行できます。この操作は元に戻せません。システム管理者のみ実行できます。

1. システム概要

ワンタッチ管理システムは、介護施設・福祉施設における設備トラブルの通報と備品管理を効率化するWebアプリケーションです。スタッフが簡単に通報でき、管理会社と連携して迅速な対応を実現します。

主要機能

  • 通報管理 — 設備トラブルの通報・追跡・履歴管理（設備名/商品名から選択、カテゴリ自動判定）
  • 管理会社連携 — 通報の自動振り分け・ステータス管理・対応実績ダッシュボード
  • 商品管理 — 設備・備品のマスタ管理、CSV/Excel/PDF/画像インポート、エクスポート
  • アカウント管理 — ロール別アクセス制御、パスワード管理
  • マスタ管理 — 会社・事業所・管理会社の階層管理

5. 認証・権限

ログインフロー

  1. 会社コード + ログインID + パスワードを入力

  3. 本番アカウント → localStorage の accounts を検索
  4. 管理会社アカウント → 会社コード欄に管理会社コード（PN001等）を入力。currentContractorもセット
  5. ロール別ホーム画面にリダイレクト
ID・パスワードは大文字小文字を区別します。

パスワード管理

  • 統一ヘッダーのメニューから「パスワード変更」
  • 現在のパスワード確認 → 新パスワード（6文字以上）→ 確認入力

6. DEMOモード

会社コード「TAMJ」でログインするとDEMOモードが有効になります。ヘッダーに紫色の「DEMOモード」バッジが表示されます。

判定ロジック（isDemoMode）

  • スタッフ・管理者: currentUser.companyCode === 'TAMJ'
  • 管理会社: currentUser.isDemoMode === true（assignedCompaniesにTAMJが含まれる場合）

7. 通報・管理会社連携

通報フロー

  1. スタッフが通報画面を開く
  2. 設備名/商品名を選択（カテゴリタブで絞り込み + 検索サジェスト）
  3. 緊急度を選択（低・中・高）
  4. 状況を入力（200文字以内）
  5. 写真を添付（最大3枚、任意）
  6. 通報ボタンで送信 → 担当管理会社に自動割り当て

管理会社自動割り当て

商品マスタのassignedPartnerIdで商品→管理会社を直接紐づけ。通報時に自動的に割り当てられます。未設定の場合は「担当未割当」の状態で保存されます。

11. 設計判断の記録

DEMOモードをsessionStorageにした理由

ブラウザを閉じるとデータが消える → デモとして適切。本番データとの混在を防ぐ。ログアウト時はデータを退避・復元（ロール切替テスト用）。

管理会社ログインを統一ログイン画面にした理由

ログイン画面が分かれていると混乱する。会社コード欄に管理会社コード（PN001等）を入力する運用で一本化。

管理会社を自動割り当てにした理由

スタッフが管理会社を選ぶ手間を省く。商品→管理会社の直接紐づけでヒューマンエラーを防ぐ。該当管理会社がいない場合は未割当で通報。

契約テーブルはインポート補助機能として位置づけた理由

CSVインポート時に管理会社列を省略できるようにするための補助機能。手動登録・編集では契約なしでも管理会社を直接選択可能。

カテゴリを介護施設向けに再設計した理由

旧カテゴリは「設備の種類」で分類していたが施設スタッフには馴染みにくい。新カテゴリは「場所・用途」ベースで直感的に選べる。

通報画面から場所欄・緊急通報を削除した理由

設備を選択すると設置場所が自動入力されるため場所欄は重複。緊急通報（火災・急病等）は119/110への直接連絡が適切で、このシステムを経由するのは不適切。

アカウントIDを自動生成にした理由

「会社コード + ログイン名」形式（例: TAMJ-tanaka）で自動生成。手入力ミスを防ぎ、QRコードへの埋め込みも容易になる。

商品マスタをテーブル表示にした理由

account-master.htmlと表示方式を統一。多項目を一覧で比較しやすい。ヘッダー固定、カテゴリバッジ、スマホ横スクロール対応。
